# Mental Model 卡片格式規範

> 此檔案是知識庫的單一真相來源（SSOT）。所有 65 張模型卡片必須符合此格式。

## 檔名規則

`vault/models/{id:02d}-{name_zh}.md`

範例：`01-供需法則.md`、`29-確認偏誤.md`、`65-地圖不等於領土.md`

特殊字元處理：
- `≠` → `不等於`
- `/` → `-`
- 其餘繁中字元、英數、`-`、`_` 可保留

## YAML Frontmatter

```yaml
---
id: 1
slug: supply-demand
name_zh: 供需法則
name_en: Supply & Demand
discipline: 經濟學
case_anchor: De Beers 鑽石
tags: [經濟學, 定價, 市場機制]
related:
  - "[[02-邊際效用遞減]]"
  - "[[06-市場均衡]]"
  - "[[14-價格歧視]]"
---
```

欄位定義：
- `id`：整數 1–65
- `slug`：英文 kebab-case，給 JSON / URL 用
- `name_zh` / `name_en`：模型名稱
- `discipline`：七大學科之一（經濟學 / 心理學 / 物理學與系統 / 生物學與演化 / 統計學 / 工程學 / 哲學與邏輯）
- `case_anchor`：主個案錨點
- `tags`：3–5 個分類標籤
- `related`：2–4 個 Obsidian wikilink，指向相關模型

## 內文結構（依序）

```markdown
# #{id} {name_zh} · {name_en}

> 學科：{discipline} ｜ 經典個案：{case_anchor}

## 一句話定義
[一句不超過 40 字的精煉定義]

## 核心原理
[2–3 段，每段 2–4 句，闡述模型的運作機制、條件、邊界。從一般商業角度切入，避免學術術語堆疊]

## 為什麼商業上重要
[1 段，3–5 句，說明為什麼經理人/創業者/投資人應該掌握這個模型]

## 商業應用場景
- **場景一**：[一句情境 + 一句說明該模型如何派上用場]
- **場景二**：...
- **場景三**：...
- **場景四**：...

## 經典個案剖析

### 主案例：{case_anchor}
[200–280 字深度分析。要交代：背景、模型如何在這個案例中體現、結果、留下什麼教訓]

### 延伸案例
- **{案例 A 名稱}**：[1–2 句說明該模型在這個案例中的作用]
- **{案例 B 名稱}**：[1–2 句]

## 實務啟示
- 啟示一（給經營者）
- 啟示二（給投資人）
- 啟示三（給創業者 / 產品負責人）
- 啟示四（通用思維）

## 常見陷阱
- **陷阱一**：[誤用或過度延伸的方式 + 怎麼避免]
- **陷阱二**：...

## 延伸思考
1. [一個能讓讀者套用到自己工作場景的問題]
2. [另一個提問]

## 相關模型
- [[XX-OOO]] — 一句話說明為什麼相關
- [[XX-OOO]] — 一句話
- [[XX-OOO]] — 一句話
```

## 風格規範

1. **語言**：繁體中文（台灣用語）。專有名詞與術語首次出現附英文，如「規模經濟（Economies of Scale）」。
2. **視角**：一般商業視角——經理人、創業者、投資人、產品負責人都讀得懂。**不要**連結到使用者的特定研究領域。
3. **個案來源**：使用真實、可查證的商業/歷史案例。避免杜撰。年份、數字若不確定就不要寫死。
4. **長度**：每張卡片內文（不含 frontmatter）控制在 800–1300 字之間。
5. **原創**：所有文字從你自己的商業知識重新撰寫。**不要**參照或復述任何特定來源。
6. **跨學科連結**：`related` 至少有一個應跨學科（例如經濟學的卡片可連結到一個心理學模型）。

## 寫作品質檢核

每張卡片寫完應自我檢查：

- [ ] 是否用一句話定義清晰交代了模型？
- [ ] 是否有具體商業情境，而非抽象概念堆疊？
- [ ] 主個案剖析是否真的進入了案例細節，而非泛泛而談？
- [ ] 實務啟示是否可操作，而非空泛口號？
- [ ] 常見陷阱是否真的有指出「人們常犯的錯」？
- [ ] 相關模型連結是否合理且有解釋為什麼相關？
